1. Overview — the 10-second tour

A user types a natural-language question (Streamlit UI or smoke harness). One entrypoint — harness_entrypoint.run_query_v10() — routes the prompt once and returns a single canonical response shape regardless of path.
  • KNOWN · ~12 IDRE API endpoints. Router signature-matches the prompt, calls the IDRE REST API, renders the body as markdown.
  • DERIVED · LangGraph 14-agent pipeline. Generates SQL via Gemini + tools, executes on MySQL, enriches + formats the result.
If a known-path API call fails (e.g. IDRE unreachable), the entrypoint gracefully falls back to the derived path with the error preserved in agent_trace.

4. Known path — IDRE API call

Sequence

  1. route(prompt) returns {path:"known", report, parameters, confidence}
  2. IdreApiClient()._auto_session() calls /api/dev/auto-login DEV-ONLY — needs production-auth rewrite
  3. client.call(report, parameters) → HTTP GET to {IDRE_BASE_URL}{endpoint}
  4. _known_path_response() normalizes the body via _coerce_rows()
  5. Markdown rendered via _render_known_markdown() (mirrors response_formatter style)
  6. Trace entries: Router + IDRE API Client with status, latency
  7. Returned dict has the canonical shape (see §2)

Body-shape normalization (_coerce_rows)

IDRE returns_coerce_rows produces
[{a:1}, {a:2}]same (list of row-dicts)
{data: [...], meta: ...}[...] (unwrapped)
{kpi: 42} (single scalar)[{kpi: 42}] (wrapped)

Failure path (graceful fallback to derived)

try:
    resp = client.call(rd.report, rd.parameters)
except Exception as e:
    # Connection refused / auth-fail / 5xx → fall back to derived
    return _run_derived(prompt, rd, ..., error=str(e))

8. Retry & abort logic

Validator and executor errors route to the Debugger, which classifies the error against 14 regex patterns and chooses a retry_strategy. The orchestrator's _route_after_debugger honours the strategy.

Debugger error patterns (14, after Node 6.1)

Error typeTriggerStrategy
HALLUCINATED_COLUMN"Unknown column 'X'"rewrite
HALLUCINATED_TABLE"Table 'db.X' doesn't exist"rewrite
HALLUCINATED_TABLE"Unknown table(s) referenced: [X]"rewrite
HALLUCINATED_COLUMN"Hallucinated column(s) detected ... Column `X`.`Y`"rewrite
AMBIGUOUS_COLUMN"Column 'X' in field list is ambiguous"rewrite
SYNTAX_ERROR"You have an error in your SQL syntax"rewrite
TIMEOUT"maximum statement execution time exceeded"suggest_filter
DIVISION_BY_ZERO"Division by zero"rewrite

WRONG_STATEMENT_TYPE"Query must be a SELECT statement"abort
LOCK_TIMEOUT"Lock wait timeout exceeded"retry_as_is
NO_SQL_EMITTED Node 6.1"Empty SQL generated."rewrite (specific instruction)
UNKNOWNregex miss; LLM-classify (attempt ≥ 2)rewrite

Per-attempt escalation

Attempt 1
Use the per-pattern fix instruction as-is
Attempt 2
Append: "Simplify query, remove optional columns, reduce JOINs"
Attempt 3 (final)
Append: "Minimum-viable query: 2 tables max, simple WHERE, no subqueries/HAVING/complex aggregates"

State updates the debugger writes

{
  "retry_context": "RETRY ATTEMPT N — ERROR ANALYSIS: ... Fix instruction: ...",
  "debug_error_type": "WRONG_STATEMENT_TYPE",
  "debug_retry_strategy": "abort",     // honoured by _route_after_debugger
  "execution_error": None,             // cleared so downstream doesn't re-trigger
  "error_message": "",
}

Knowledge base learning loop

data/error_knowledge_base.json — the debugger writes a candidate fix on every classification (success: None). After the next retry, _track_fix_result calls mark_fix_success(error_type, success=bool). Future debugger runs on the same error_type can reference what worked before via _kb_lookup (returns successful fixes only).
